## El ciclo SDD

SDD organiza el desarrollo en cuatro fases secuenciales. Cada fase produce un artefacto que alimenta la siguiente.

<Steps>

1. **Explorar**

   Entender el contexto antes de proponer soluciones. Claude lee el código existente, identifica patrones, dependencias y restricciones.

   **Plantilla:**
   ```
   Explora [área del codebase]. Necesito entender:
   - La estructura actual de [componente/módulo]
   - Las dependencias relevantes
   - Los patrones utilizados
   NO hagas ningún cambio todavía. Solo reporta lo que encuentres.
   ```

2. **Planificar**

   Diseñar la solución antes de implementarla. Se usa Plan Mode para obtener aprobación explícita.

   **Plantilla:**
   ```
   Basándote en tu exploración, diseña un plan para [objetivo].
   El plan debe incluir:
   - Archivos a modificar y por qué
   - Enfoque técnico elegido y alternativas descartadas
   - Posibles riesgos o efectos secundarios
   Espera mi aprobación antes de implementar.
   ```

3. **Implementar**

   Ejecutar el plan aprobado con cambios precisos y verificables.

   **Plantilla:**
   ```
   Implementa el plan aprobado. Para cada cambio:
   - Haz ediciones quirúrgicas (no reescrituras innecesarias)
   - Ejecuta los tests después de cada paso significativo
   - Detente si encuentras algo inesperado y repórtalo
   ```

4. **Commit**

   Cerrar el ciclo con un registro claro del cambio realizado.

   **Plantilla:**
   ```
   Crea un commit con los cambios implementados.
   El mensaje debe explicar el QUÉ y el POR QUÉ,
   no solo el CÓMO. Sigue la convención del proyecto.
   ```

</Steps>

## Ejemplo: spec.md para un componente real

La Fase 1 (Spec) produce un archivo como este:

```markdown
# Spec: Componente LoginForm

## Contexto
Formulario de inicio de sesión para el portal interno TIC.
Reemplaza el formulario actual en /app/login/page.tsx.

## Criterios de aceptación
- [ ] Campos: email (validación RFC 5322) y contraseña (mínimo 8 caracteres)
- [ ] Estado de carga visible al hacer submit
- [ ] Mensaje de error específico si las credenciales son incorrectas
- [ ] Mensaje de error genérico si hay falla de red
- [ ] Accesible: etiquetas ARIA, navegación por teclado, foco visible
- [ ] Responsive: funciona en móvil (320px) y escritorio

## Restricciones técnicas
- Usa componentes de shadcn/ui (Input, Button, Form)
- Integra con el hook useAuth existente en src/hooks/auth.ts
- No crear nuevas dependencias npm

## Lo que NO incluye este spec
- Registro de usuarios / recuperación de contraseña (sprint futuro)

## Definición de "terminado"
- Tests unitarios con >80% cobertura
- Pasa npm run lint sin errores
- Revisión de PR aprobada
```

**El spec es el contrato, no el prompt.** El spec persiste en el repositorio, vive con el código y puede ser revisado en un PR. El prompt es efímero.

## Por qué las 4 fases importan

Saltarse fases es la causa más común de resultados insatisfactorios con IA. La exploración sin plan lleva a implementaciones que no encajan. La implementación sin exploración introduce regresiones. SDD hace cada fase **explícita y verificable**.
