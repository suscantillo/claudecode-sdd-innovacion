## Contexto

El equipo de diseño entregó wireframes en Figma para un formulario complejo de onboarding con múltiples pasos, validaciones y lógica condicional. El desarrollador necesita convertirlo en código sin perder detalles.

Flujo completo: wireframe → spec → design.md → generación con Claude Code.

## Flujo de trabajo

### 1. Describir el wireframe con precisión

En lugar de adjuntar la imagen y pedir "haz esto", describir el componente en detalle:

```
Necesito crear un formulario de onboarding multi-paso con shadcn/ui.

Especificaciones:
- 3 pasos: Datos personales → Empresa → Preferencias
- Paso 1: nombre (requerido), email (requerido, validar formato),
  teléfono (opcional)
- Paso 2: nombre empresa (requerido), tamaño (select: 1-10, 11-50,
  51-200, 200+), industria (select con opciones predefinidas)
- Paso 3: rol (radio: Developer/Designer/Manager/Other),
  si Other → mostrar campo de texto adicional (condicional)
- Navegación: Anterior/Siguiente con validación por paso
- Estado de progreso visual (stepper)

El proyecto ya usa: react-hook-form, zod, shadcn/ui.
Sigue los patrones existentes en src/components/forms/.
```

### 2. Explorar primero

```
Antes de crear el componente, explora src/components/forms/
para entender los patrones existentes de formularios y validación.
```

Claude identifica que ya existe un patrón con `FormField`, `FormItem`, `FormLabel` de shadcn y que los schemas Zod se definen en archivos separados `*.schema.ts`.

### 3. Plan del componente

```
/plan

Diseña la estructura del componente OnboardingForm:
- Qué archivos crear
- Cómo dividir los pasos en sub-componentes
- Dónde definir el schema Zod
- Cómo manejar el estado del wizard
```

**Plan aprobado:**
```
src/components/onboarding/
├── OnboardingForm.tsx          # Componente principal con stepper
├── steps/
│   ├── PersonalInfoStep.tsx
│   ├── CompanyStep.tsx
│   └── PreferencesStep.tsx
└── onboarding.schema.ts        # Schema Zod completo
```

### 4. Implementar paso a paso

```
Implementa el plan. Empieza por el schema Zod,
luego el componente principal y después cada paso.
Verifica TypeScript después de cada archivo.
```

## Resultado

Componente funcional en ~45 minutos, siguiendo exactamente los patrones del proyecto, con validación por paso y lógica condicional correctamente implementada.

## Ejemplo adicional: UserProfileCard con spec y design.md

Para componentes con estados visuales complejos, la guía recomienda un `design.md` junto al `spec.md`:

**spec.md:**
```markdown
## Props
interface UserProfileCardProps {
  user: {
    name: string
    email: string
    role: string
    department: string
    avatarUrl?: string
    isOnline: boolean
  } | null   // null = estado loading
}

## Criterios de aceptación
- [ ] Muestra: avatar, nombre, email, rol, departamento
- [ ] Indicador online/offline sobre el avatar (verde / gris)
- [ ] Sin foto: iniciales con color determinístico (hash del nombre, no random)
- [ ] Estado loading: skeleton con animación pulse
- [ ] "Ver perfil" → navega a /perfil
- [ ] "Cerrar sesión" → llama a signOut() del contexto de auth

## Design tokens
- Avatar: 48x48px, border-radius full
- Indicador: 10x10px, verde = online-500, gris = neutral-400
- Card: bg-card, border, rounded-lg, p-4
```

**Prompt de generación:**
```
Lee spec.md y design.md en docs/specs/user-profile-card/.
Revisa cómo usamos shadcn/ui en otros componentes (especialmente
los que tienen Skeleton).
Implementa UserProfileCard en src/components/sidebar/user-profile-card.tsx
con sus tests. Sigue exactamente el spec.
```

## Clave del éxito

La especificación detallada eliminó las ambigüedades. Claude no tuvo que adivinar:
- Qué librería de formularios usar (react-hook-form + zod)
- Cómo estructurar los archivos (siguió el patrón explorado)
- Cómo manejar el campo condicional (especificado explícitamente)
- Qué design tokens usar (definidos en el spec)
