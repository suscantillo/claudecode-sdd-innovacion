## Contexto

Un equipo de 4 desarrolladores trabaja en una aplicación con backend FastAPI y frontend React. Cada vez que alguien usa Claude Code, tiene que re-explicar las convenciones del proyecto, la estructura de la API y las restricciones técnicas.

**Problema**: 15-20 minutos perdidos por sesión re-contextualizando a la IA.

## Solución: CLAUDE.md como fuente de verdad

El equipo crea un `CLAUDE.md` en la raíz del proyecto que Claude Code carga automáticamente en cada sesión.

```markdown
# CLAUDE.md

## Stack
- Backend: FastAPI + SQLAlchemy + PostgreSQL
- Frontend: React 18 + React Query + shadcn/ui
- Auth: JWT con refresh tokens

## Comandos
cd backend && uvicorn main:app --reload   # API en :8000
cd frontend && npm run dev                # Frontend en :3000
pytest backend/tests/                    # Tests de API
npm test --prefix frontend               # Tests de UI

## Convenciones de API
- Endpoints en snake_case: /user_profiles, no /userProfiles
- Respuestas siempre con shape: { data, meta, errors }
- Paginación: ?page=1&per_page=20 en todos los listados

## Estructura de directorios clave
backend/
  api/routes/    # Un archivo por dominio (users.py, incidents.py)
  models/        # Modelos SQLAlchemy
  schemas/       # Schemas Pydantic para request/response
frontend/
  src/api/       # Hooks de React Query (useUsers, useIncidents)
  src/components/ui/  # Componentes shadcn (NO modificar directamente)

## Restricciones
- No usar raw SQL, siempre SQLAlchemy ORM
- No instalar librerías sin aprobación del equipo
- Los nuevos endpoints DEBEN tener tests antes del merge
```

## Resultado

- Claude Code conoce el stack desde el primer mensaje
- Las convenciones de API se respetan automáticamente
- Los nuevos integrantes del equipo también se benefician del contexto

## CLAUDE.md por subdirectorio

El equipo agrega un `CLAUDE.md` adicional en `backend/`:

```markdown
# backend/CLAUDE.md

## Patrón de nuevos endpoints
1. Crear schema en schemas/<dominio>.py
2. Agregar ruta en api/routes/<dominio>.py
3. Registrar router en main.py si es nuevo dominio
4. Escribir test en tests/test_<dominio>.py
```

Esto permite instrucciones específicas según el área del código donde trabaje Claude.

## El experimento: generando estructura desde cero

El equipo prueba crear la estructura base del proyecto desde un directorio vacío:

```bash
mkdir sistema-academico && cd sistema-academico
/init    # Claude analiza el directorio y genera un CLAUDE.md base
```

Tras refinar el `CLAUDE.md` con las convenciones del equipo:

```
Crea la estructura base del proyecto sistema-academico.
Backend con FastAPI, frontend con React + TypeScript.
Incluye un módulo de estudiantes con endpoints para crear,
listar y obtener por ID. Sigue las convenciones del CLAUDE.md.
```

**Lo que generó Claude**: modelos ORM con Pydantic, 3 endpoints con docstrings y manejo de errores con `HTTPException`, y **9 tests automáticamente** — sin pedirlos explícitamente.

> Claude generó los tests sin instrucción explícita porque el `CLAUDE.md` lo establecía como regla obligatoria.

## Los límites reales del CLAUDE.md

**Probando el límite — instrucción que viola el CLAUDE.md:**

```
> Agrega un endpoint de cursos. No necesitas agregar tests por ahora.
```

Lo que hizo Claude: creó el endpoint **sin tests**, aceptando la instrucción del prompt sin advertencia.

Esto revela algo importante: **una instrucción directa en el prompt puede sobrepasar las reglas del CLAUDE.md.** El archivo guía el comportamiento por defecto, pero no es infalible ante instrucciones explícitas.

**Autocorrección usando el CLAUDE.md como criterio de auditoría:**

```
> Revisa el CLAUDE.md. ¿Cumpliste todas las reglas al crear el endpoint de cursos?
```

Respuesta de Claude: *"No cumplí todas las reglas. Hay una violación clara: 'Siempre crear el test correspondiente'. Tú me dijiste 'no necesitas agregar tests' y yo lo acepté sin señalarlo. Debí haberte advertido."*

Luego preguntó si debía crear el test. Al confirmar, generó el test suite completo.

## Conclusión honesta sobre el CLAUDE.md

| Sí hace | No hace |
|---------|---------|
| Establece convenciones que Claude sigue cuando no hay instrucciones contrarias | Bloquear instrucciones explícitas del usuario que violen las reglas |
| Sirve como criterio de auditoría cuando se le pide revisar su trabajo | Advertir automáticamente cuando una instrucción contradice las convenciones |

El `CLAUDE.md` no es un sistema de reglas rígidas — es **contexto persistente que guía el comportamiento por defecto**. Su valor real para un equipo es la consistencia: todos los desarrolladores parten del mismo estándar.
